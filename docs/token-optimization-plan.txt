Problem
We want to reduce context/token usage dynamically without sacrificing quality, using proven strategies from opencode/crush and fitting conduit's architecture.

Approach
Prioritize high-impact, low-risk changes that reduce tool-result payloads and improve compaction quality, then add model-aware thresholds and metrics for adaptive tuning.

Todos
1. Add truncate-to-disk for large tool outputs (bash/read/grep/webfetch), returning a preview plus a hint to use Grep/Read with offset/limit; include retention cleanup.
2. Harden FileRead with binary detection and long-line truncation to avoid token blowups on binaries or minified files.
3. Replace compaction prompt with anchored, structured summary template and support merging <previous-summary> blocks.
4. Implement smart tool_result pruning during compaction (protect critical tools, keep recent N, truncate old; never truncate error-heavy outputs).
5. Add model-aware overflow detection to trigger micro-compact/compact based on usable context window and reserved output buffer.
6. Make truncation and compaction limits configurable (max_lines, max_bytes, keep_recent, reserved buffer) with safe defaults.
7. Add token-savings metrics (rtk, microcompact, truncation) surfaced in stats/TUI for feedback loops.
8. Update STATUS.md for newly completed capabilities and add/adjust tests for truncation/compaction behavior.

Notes
- Favor preserving errors/structured outputs; truncation must never hide failures.
- Prefer incremental roll-out: Phase 1 (todos 1-3), Phase 2 (4-6), Phase 3 (7-8).
